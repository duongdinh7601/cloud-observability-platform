# Log Service

The Log Service is the first backend service in the Cloud Observability Platform. It handles ingestion, storage, filtering, and retrieval of log entries.

---

## Responsibilities

- Accept log entries via `POST /logs`
- Store logs in PostgreSQL
- Store optional structured metadata in a PostgreSQL `JSONB` column
- Return logs through `GET /logs`
- Support filtering by level, service name, and time range
- Support stable cursor pagination ordered by newest logs first
- Expose readiness and liveness endpoints for container and Kubernetes health checks

---

## Data Model

The current SQLAlchemy model maps to the `logs` table.

| Field | Type | Description |
| --- | --- | --- |
| `id` | integer | Primary key generated by PostgreSQL |
| `timestamp` | timezone-aware datetime | Log event timestamp |
| `level` | string | Log severity such as `INFO`, `WARNING`, `ERROR`, or `DEBUG` |
| `service_name` | string | Service that produced the log |
| `message` | string | Human-readable log message |
| `metadata` | JSONB, nullable | Optional structured context such as request IDs, durations, or tenant information |

The Python ORM attribute for the database `metadata` column is `log_metadata` to avoid colliding with SQLAlchemy's internal `metadata` attribute. The external API still exposes the field as `metadata`.

## API Endpoints

Handled API responses include an `X-Request-ID` header. If a request provides `X-Request-ID`, the service preserves it; otherwise the service generates one. The same value appears in structured operational request logs so a response can be correlated with backend behavior.

### `POST /logs`

Creates a new log entry.

Request body:

```json
{
  "timestamp": "2026-01-09T18:00:00Z",
  "level": "ERROR",
  "service_name": "frontend",
  "message": "Failed to fetch logs",
  "metadata": {
    "request_id": "req-123",
    "duration_ms": 67
  }
}
```

Response body:

```json
{
  "id": 1,
  "timestamp": "2026-01-09T18:00:00Z",
  "level": "ERROR",
  "service_name": "frontend",
  "message": "Failed to fetch logs",
  "metadata": {
    "request_id": "req-123",
    "duration_ms": 67
  }
}
```

Response codes:

- `201` - log entry created successfully
- `422` - invalid request body
- `500` - internal server error

### `GET /logs`

Retrieves log entries with optional filters and cursor pagination.

Query parameters:

| Field | Type | Description |
| --- | --- | --- |
| `limit` | integer | Number of logs to return, from `1` to `200`; defaults to `50` |
| `cursor_ts` | datetime | Timestamp component of the cursor for the next page |
| `cursor_id` | integer | ID component of the cursor for the next page |
| `level` | string | Filter by log severity; `WARN` is normalized to `WARNING` |
| `service_name` | string | Case-insensitive service-name filter |
| `start_time` | datetime | Earliest timestamp to include |
| `end_time` | datetime | Latest timestamp to include |

Request example:

```text
GET /logs?level=ERROR&service_name=frontend&limit=20
```

Response example:

```json
{
  "items": [
    {
      "id": 1,
      "timestamp": "2026-01-09T18:00:00Z",
      "level": "ERROR",
      "service_name": "frontend",
      "message": "Failed to fetch logs",
      "metadata": {
        "request_id": "req-123"
      }
    }
  ],
  "next_cursor": {
    "cursor_ts": "2026-01-09T18:00:00Z",
    "cursor_id": 1
  }
}
```

Response codes:

- `200` - logs retrieved successfully
- `422` - invalid query parameters
- `500` - internal server error

Notes:

- Logs are ordered by `timestamp DESC, id DESC`.
- The cursor is composed of `(timestamp, id)` so pagination remains stable when multiple logs share the same timestamp.
- If one cursor field is provided, both `cursor_ts` and `cursor_id` are required.
- `start_time` must be less than or equal to `end_time` when both are provided.

## Database Migrations

The service uses Alembic for schema changes. Migration files live under:

```text
services/log-service/migrations/
```

The FastAPI app does not call `Base.metadata.create_all()` on startup. Schema changes are applied deliberately through Alembic migrations.

Current migration history:

- `2e3e3bad4dcc_create_logs_table.py` creates the initial `logs` table and indexes.
- `cc18b0f81e9b_add_log_metadata.py` adds nullable `metadata` JSONB storage.

Future production rollout should run migrations in a controlled CI/CD step, likely through a Kubernetes Job using the same `log-service` image tag as the app rollout.

## Operational Logs

The service emits structured JSON logs about its own HTTP request handling. These are operational logs from the platform service, not product log entries stored in the `logs` table.

Current behavior:

- non-health HTTP requests produce one JSON request log
- health endpoints are skipped to reduce Kubernetes probe noise
- request logs include `event`, `request_id`, `method`, `path`, `status_code`, and `duration_ms`
- unhandled exceptions produce an `http_request_error` log with error type and message before the exception is re-raised
- handled responses include `X-Request-ID`

Example request log:

```json
{
  "duration_ms": 12.34,
  "event": "http_request",
  "method": "GET",
  "path": "/logs",
  "request_id": "req-123",
  "status_code": 200
}
```

Current production-shaped follow-ups:

- replace the middleware-local logger handler with centralized service-wide JSON logging configuration
- consider filtering noisy paths such as `/favicon.ico` and CORS `OPTIONS` preflight requests
- add request logging tests when request ID behavior becomes a stronger contract
- decide later whether the platform should ingest its own operational logs

## Testing

The integration tests use a dedicated test database.

Test setup:

- checks that `TEST_DATABASE_URL` is configured
- refuses destructive reset work unless the URL looks like a test database
- drops test schema objects such as `logs` and `alembic_version`
- runs Alembic migrations to `head`
- overrides the FastAPI database dependency so API tests use the test database

Current tests verify:

- empty log-list behavior
- cursor pagination
- metadata round-tripping through `POST /logs` and `GET /logs`

Future test infrastructure improvements:

- parse the test database URL and require an exact expected database name
- optionally require an explicit reset flag such as `ALLOW_TEST_DB_RESET=true`
- use disposable per-run databases or containers in CI
